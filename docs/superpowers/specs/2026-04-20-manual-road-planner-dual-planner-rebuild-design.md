# 手动道路规划器双规划器重建设计

> 日期: 2026-04-20
> 范围: manual road planner / preview UI / bridge generation / demolition targeting
> 前置: `2026-04-20-road-bugfix-design.md`、`2026-04-19-road-system-strengthen-design.md`

## 1. 概述

当前手动道路规划器存在一组互相关联的问题：

- 方案选择后会短暂弹出路宽选择，但随后又被预览包拉回方案页
- `detour` 和 `bridge` 实际共用同一路径逻辑，只是标签不同
- `bridge` 没有真正的“走桥优先”策略，海岛连大陆常直接失败
- 预览和锚点选择仍会把树冠、树干等上层植被当地表
- `both` 模式没有并行展示两种明确不同的候选
- 短跨水面/峡谷应允许悬空桥面，但长跨越应改成桥墩大桥
- 道路拆除要求过于精确，导致玩家难以锁定整条道路
- 配置页按钮只是在本地切屏或把配置存到服务端，没有真正参与当前候选预览和最终生成

本设计把手动道路规划器改成“两个真实规划器 + 一个编排器”的结构：

- `detour planner`: 陆路优先，只允许短跨无桥墩
- `bridge planner`: 独立的桥梁优先规划器，主动找桥头和跨越走廊
- `both orchestrator`: 并行跑前两者，向客户端展示两个真实候选

同时统一修复：

- 真实地表采样
- UI 状态机和按钮接线
- 配置回填与预览重建
- 宽容式道路拆除锁定

## 2. 目标行为

### 2.1 方案语义

- `detour` 代表“尽量绕陆路”
- `bridge` 代表“主动走桥”
- `both` 代表“同时给出 detour 和 bridge 两个候选”

任何模式都不能再退化成“同一条路径的两个名字”。

### 2.2 短跨规则

- 连续跨越长度 `<= 8` 格时，允许生成 `short-span`
- `short-span` 桥面下方悬空，不打桥墩
- `short-span` 同时适用于水面和峡谷/悬崖空洞

### 2.3 长跨规则

- 连续跨越长度 `> 8` 格时，进入 `pier-bridge`
- `pier-bridge` 使用桥头平台 + 长桥面 + 大量桥墩
- `pier-bridge` 只由 `bridge planner` 主动生成

### 2.4 海岛规则

- 海岛连大陆只能依赖 `bridge planner`
- `bridge planner` 成功即可在 `both` 中给出候选
- 不允许因为 `detour` 失败就把整体结果判为失败

### 2.5 拆除规则

- 保留拆除模式
- 改成“看向道路附近即可锁定整条路”
- 不再要求玩家精确命中中心线路径坐标

## 3. 总体架构

### 3.1 公共层

以下能力改成公共层，由所有 planner 共享：

- 真实地表采样
- 锚点合法性判断
- 预览缓存
- 配置缓存
- UI 状态机
- 拆除目标锁定

### 3.2 `detour planner`

`detour planner` 保持现有陆路寻路管线为核心，但严格限制桥化能力：

- 优先走实地和缓起伏地形
- 对长连续水域和峡谷保持高惩罚
- 仅当绕行代价明显更高且跨越长度 `<= 8` 时，允许直接生成 `short-span`
- 对 `> 8` 的连续跨越，直接放弃该跨越并继续绕行

因此，`detour` 允许“小桥补洞”，但不允许退化成大桥方案。

### 3.3 `bridge planner`

`bridge planner` 独立实现，不再先跑 detour 再补桥。

它的流程是：

1. 搜索起点和终点附近的岸线/峡谷边缘桥头锚点
2. 枚举可用桥头配对
3. 评估跨水或跨峡谷走廊
4. 选出桥优先中心线
5. 根据连续跨越长度分级成 `short-span` 或 `pier-bridge`
6. 在桥两端补接地道路

这让 `bridge` 的主体始终是“桥先行，陆地接驳”，而不是“陆路先行，碰到水再凑桥”。

### 3.4 `both orchestrator`

`both` 不再是第三套独立逻辑，只承担编排职责：

- 并行请求 `detour planner`
- 并行请求 `bridge planner`
- 收集成功候选和失败原因
- 把真实结果推给客户端

展示规则：

- 两个都成功: 展示两个候选
- 只有一个成功: 只展示一个候选
- 两个都失败: 返回整体失败

## 4. 地表采样与锚点规则

### 4.1 统一真实地表采样

所有以下逻辑统一改用“忽略植被/树冠/可替换方块”的真实地表采样：

- `surfaceAt(...)`
- 城镇核心地表定位
- 桥头锚点搜索
- 边界候选搜索
- 岛屿桥头搜索
- 预览路径的高度回写

禁止继续直接使用 `Heightmap.Types.WORLD_SURFACE` 作为锚点地表来源。

### 4.2 需要忽略的上层方块

采样时应忽略至少以下类型：

- 树叶
- 原木
- 花草
- 藤蔓
- 雪层
- 可替换植物
- 竹子、甘蔗、浆果灌木
- 巨型蘑菇相关方块

目标是把“能站得住并承载路面/桥头”的方块当作真实地表。

### 4.3 桥头锚点规则

`bridge planner` 的桥头锚点优先级：

1. 靠近连续水域或峡谷边缘的真实地表
2. 与目标方向一致的岸线点
3. 与已有道路可衔接的岸线点
4. 退化到边界地表点

海岛场景中，如果普通边界锚点不满足条件，必须切换到岸线优先搜索。

## 5. `bridge planner` 详细规则

### 5.1 桥头候选搜索

每端搜索一组桥头候选，而不是单点锚定。

候选打分应考虑：

- 是否位于真实地表
- 是否靠近连续跨越边界
- 与另一端的直连方向是否稳定
- 是否能在桥外侧补出短接地道路
- 是否避开核心禁区和明显阻挡结构

### 5.2 跨越走廊打分

`bridge planner` 需要主动偏好以下走廊：

- 海岛到大陆的连续跨水走廊
- 两岸之间明显比绕路更短的跨水走廊
- 两侧边缘明确的峡谷直连走廊

负面打分因素：

- 过多不必要弯折
- 桥头离城镇出口过远
- 走廊被陆地频繁切断
- 走廊会穿过禁区或结构密集区

### 5.3 桥型分级

`bridge planner` 输出的桥梁分两类：

#### `short-span`

- 触发条件: 连续跨越长度 `<= 8`
- 桥面直接悬空
- 无桥墩
- 两端仍允许有桥头过渡

#### `pier-bridge`

- 触发条件: 连续跨越长度 `> 8`
- 使用桥头平台
- 使用长桥面
- 大量使用桥墩
- 峡谷和水面走同一套分级逻辑

### 5.4 海岛连大陆

海岛模式下：

- `bridge planner` 允许桥头直接落在岸线边缘，而不是必须先找到陆路出口
- 优先选择跨水连续性最强的走廊
- 如果桥本体成功，不要求 `detour planner` 也成功

## 6. `detour planner` 详细规则

### 6.1 总体原则

- 陆路优先
- 绕水优先
- 绕峡谷优先
- 小跨越可补桥
- 大跨越不建桥

### 6.2 小跨越补桥

当 detour 遇到以下情况时允许 `short-span`：

- 连续水面长度 `<= 8`
- 连续峡谷/空洞长度 `<= 8`
- 绕行代价明显高于直接短跨

### 6.3 禁止升级为大桥

一旦跨越长度 `> 8`：

- detour 不得切换到 `pier-bridge`
- detour 必须继续尝试绕行
- 绕行失败则该 detour 候选失败

## 7. UI 状态机与按钮逻辑

### 7.1 状态机

新的手动道路流程：

1. 目标选择
2. 服务端异步规划
3. 方案选择
4. 配置页
5. 服务端根据“当前方案 + 当前配置”重建预览
6. 预览确认
7. 正式生成

### 7.2 方案页按钮

`Use Option` 按钮的问题是：

- 当前实现会先本地切到配置页
- 服务端随后再下发预览包
- 客户端看到两个候选后又自动重新打开方案页

修正后：

- `Use Option` 只发送“切换当前 option”的意图
- 方案页切到配置页时要记录“当前处于配置阶段”
- 普通预览刷新不得自动重新打开方案页
- 只有“规划首次完成”事件才能自动打开方案页

### 7.3 配置页按钮

`Confirm` 目前只存配置并关闭界面，逻辑错误。

修正后：

- `Confirm` 要触发“用当前 option + 新配置重建预览”
- 客户端显示等待中的配置态
- 服务端返回新的 preview/build cache 后，客户端回到预览确认态

`Cancel` 修正后：

- 返回上一步的方案/预览上下文
- 不清空当前候选
- 不把玩家直接踢回游戏主视图

### 7.4 配置回填

配置页打开时必须回填当前配置：

- 宽度
- 材质
- 隧道开关

已选按钮应表现为“已选中”而不是单纯 disabled。

### 7.5 目标页滚轮

滚轮事件只应在鼠标位于列表区域时吞掉，避免误拦截。

## 8. 预览缓存与数据流

### 8.1 两层缓存

服务端需要区分：

#### `route candidate cache`

缓存内容：

- 当前目标 town
- detour candidate
- bridge candidate
- 每个候选的中心线
- 桥型分段
- 桥头信息
- 每个候选的失败原因

#### `build preview cache`

缓存内容：

- 当前选中的 option
- 当前配置
- 按配置重新构建后的 ghost blocks
- bridge ranges
- preview hash
- 最终确认所需 build steps

### 8.2 网络事件分离

协议语义拆分为以下几类事件：

- `PlanningStarted`
- `PlanningResult`
- `SelectOption`
- `ApplyConfig`
- `PreviewUpdated`
- `ConfirmPreview`

要求：

- 规划结果事件可以开方案页
- 预览刷新事件不能抢屏
- 配置应用和方案选择是两个独立步骤

## 9. 拆除锁定重写

### 9.1 当前问题

当前拆除依赖精确命中一个道路表面方块，并且要求命中点与记录路径的 `x/z` 高度一致，导致极难使用。

### 9.2 新锁定规则

拆除改成宽容式锁定：

1. 获取玩家视线命中点
2. 在命中点周围半径内搜索候选道路
3. 候选来源同时包括：
   - 道路中心线
   - 已建路面方块
   - 桥面方块
4. 对候选整条路打分：
   - 距离视线命中点越近越优先
   - 与玩家视角方向越一致越优先
   - 命中桥面或路面时整条路优先级提升
5. 锁定得分最高的整条道路

### 9.3 拆除交互

- 第一次右键: 高亮整条路并提示确认
- 第二次右键: 执行整条路拆除
- 若锁定目标已失效: 清空锁定并提示目标失效

## 10. 失败处理

### 10.1 `both`

- 任一 planner 成功即可进入候选展示
- 两个都失败才返回整体失败

### 10.2 `bridge`

需要区分失败原因：

- 找不到桥头
- 找不到连续跨越走廊
- 走廊被禁区阻断
- 桥外接地段无法落地

`bridge` 失败时不得静默退化成 `detour`。

### 10.3 `detour`

需要区分失败原因：

- 陆路不可达
- 长跨越不可绕行
- 短跨补桥后仍不可达

### 10.4 `demolish`

需要区分：

- 未锁定道路
- 道路已失效
- 拆除执行失败

## 11. 验证

### 11.1 自动验证

至少增加以下逻辑测试：

- `bridge planner` 海岛桥头搜索
- `bridge planner` 短跨/长跨分级
- `detour planner` 仅允许 `<= 8` 的短跨
- `both orchestrator` 的候选合并逻辑
- 配置应用后预览重建逻辑
- 宽容式拆除锁定

### 11.2 UI 回归

至少覆盖：

- 选择 option 后不会被预览刷新拉回方案页
- 配置页 confirm 会生成新的预览
- 配置页 reopen 会回填当前配置
- 只有首次规划完成才自动打开方案页

### 11.3 人工场景验证

至少验证以下 5 类场景：

- 林地树冠边缘
- 小河沟短跨
- 长水面跨越
- 峡谷跨越
- 海岛连大陆

- `.\gradlew.bat compileJava`

## 12. 实施顺序

推荐实施顺序：

1. UI 状态机与按钮接线修正
2. 真实地表采样统一
3. `bridge planner` 独立规划链路
4. `detour planner` 收紧为“短跨可补、长跨不桥化”
5. `both orchestrator` 改成双候选编排
6. 配置接入 preview/build cache
7. 拆除锁定重写
8. 测试与验证
